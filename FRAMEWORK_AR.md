# إطار المشروع — منصة الألعاب العُمانية للأطفال

**الإصدار:** 1.0  
**التاريخ:** 2026-07-21  
**الغرض:** مخطط تقني لتحويل النموذج الأولي إلى منصة ألعاب ويب جاهزة للإنتاج تضم 20 لعبة عُمانية الثيم للأطفال.

---

## 1. الرؤية والنطاق

بناء منصة ألعاب ويب آمنة، عالية الأداء، وموثوقة ثقافيًا للأطفال في الفئة العمرية 7–12 عامًا. تدعم المنصة:

- ألعاب اللاعب الفردي وألعاب 1 ضد 1 عبر الإنترنت.
- واجهة عربية أولية مع دعم كامل للتخطي من اليمين إلى اليسار (RTL).
- تواصل آمن للأطفال (تفاعلات جاهزة فقط).
- اللعب عبر الأجهزة على الكمبيوتر والجهاز اللوحي.
- الاحتفاظ بالنتائج، ملفات اللاعبين البسيطة، ولوحات المتصدرين.

## 2. المكدس التقني المقترح

### 2.1 الواجهة الأمامية الأساسية

| الطبقة | التقنية | السبب |
|--------|---------|-------|
| أداة البناء | **Vite** | خادم تطوير سريع، إعداد بسيط، إضافة PWA، دعم ممتاز لـ TypeScript. |
| اللغة | **TypeScript** | أمان الأنواع لمنطق اللعبة، رسائل الشبكة، والحالة. يقلل الأخطاء مع نمو الكتالوج. |
| هيكل التطبيق | **Svelte 5** أو **Vue 3** | خفيف، تفاعلي، دعم ممتاز لـ RTL والتدويل. |
| محرك الألعاب | **Phaser 3** | محرك 2D ناضج مع الفيزياء، المشاهد، تجريد الإدخال، الصوت، تأثيرات الكاميرا، وأداء مثبت على الجوال. |
| التنسيق | خصائص CSS المخصصة + **Tailwind CSS** اختياريًا | الحفاظ على لوحة الألوان العُمانية الحالية؛ إضافة فئات utility لتطوير واجهة أسرع. |

### 2.2 الخلفية والألعاب الجماعية

| الطبقة | التقنية | السبب |
|--------|---------|-------|
| خادم اللعبة | **Colyseus** | إطار عمل للألعاب الجماعية مع مخططات TypeScript، إدارة الغرف، وإعادة الاتصال. |
| قاعدة البيانات / المصادقة | **Supabase** | PostgreSQL، مصادقة، حضور فوري، ولوحات متصدرين مع طبقة مجانية سخية. |
| بديل | **Socket.io + Express + Redis** | استخدمه إذا كنت تفضل التحكم المنخفض المستوى والمنطق المخصص. |

### 2.3 الأدوات

| الفئة | الأداة |
|-------|--------|
| اختبارات الوحدات | **Vitest** |
| اختبارات end-to-end | **Playwright** |
| Linting | **ESLint** + **Prettier** |
| PWA | **إضافة Vite PWA** |

### 3.1 الفصل بين المهام

- **هيكل التطبيق** يتعامل مع التوجيه، التنقل، النوافذ المنبثقة، ملف اللاعب، إعدادات الصوت، وردهة الألعاب الجماعية.
- **مدير الألعاب** يُنشئ مشاهد Phaser، ويدير الإيقاف/الاستئناف، التنظيف، والتواصل بين الهيكل واللعبة.
- **مشاهد الألعاب** تحتوي فقط على منطق اللعبة، العرض، والإدخال المحلي.
- **طبقة الشبكة** (عميل Colyseus) تُحقن في المشاهد التي تحتاج إلى اللعب الجماعي؛ لا تتحدث المشاهد مباشرة مع الخادم.
- **الخادم** يتحقق من جميع الحركات ويملك الحالة الموثوقة.

## 5. إرشادات تطوير الألعاب

### 5.1 كل لعبة هي مشهد Phaser

```ts
// src/games/runner/RunnerScene.ts
import { Scene } from 'phaser';

export class RunnerScene extends Scene {
  constructor() {
    super({ key: 'RunnerScene' });
  }

  create() {
    // تهيئة العالم، اللاعب، العقبات
  }

  update(time: number, delta: number) {
    // حلقة اللعبة
  }
}
```

### 5.2 عقد إعداد اللعبة

كل لعبة تصدر كائن إعداد يستهلكه مدير الألعاب:

```ts
export const runnerConfig: GameConfig = {
  id: 'frankincense',
  name: 'مغامرة جامع اللبان',
  icon: '🏃‍♂️',
  supportsSingle: true,
  supportsOnline: false,
  sceneKey: 'RunnerScene',
  preloadAssets: [...],
};
```

### 5.3 مبادئ التصميم

1. **الأصالة الثقافية:** استخدم المرئيات والألوان والرموز العُمانية في الرسومات والخلفيات والمؤثرات الصوتية.
2. **الصعوبة المناسبة:** استهدف الفئة العمرية 7–12. تجنب ميكانيكيات التوقيت السريع التي تتطلب ردود فعل البالغين.
3. **جلسات قصيرة:** استهدف جلسات لعب مدتها 1–3 دقائق.
4. **تغذية راجعة واضحة:** كل إجراء يجب أن يحصل على تغذية راجعة بصرية وصوتية.
5. **اللمس أولًا:** يجب أن تكون جميع الألعاب قابلة للعب بالكامل على جهاز لوحي 10 بوصة بدون لوحة مفاتيح.

## 6. معمارية الألعاب الجماعية

### 6.1 نموذج الخادم الموثوق

الخادم هو المصدر الوحيد للحقيقة. يرسل العملاء المدخلات/النوايا، يتحقق الخادم منها، يحدث الحالة، ويبث التغييرات.

```
العميل A          غرفة Colyseus          العميل B
   │                     │                    │
   │── move(index) ─────►│                    │
   │                     │── validate move ──►│
   │                     │◄── valid ──────────│
   │                     │                    │
   │◄── state update ────│── state update ───►│
```

### 6.2 مثال على مخطط Colyseus: Tic-Tac-Toe

```ts
// server/schemas/TicTacToeState.ts
import { Schema, type, ArraySchema } from '@colyseus/schema';

export class TicTacToeState extends Schema {
  @type(['string'])
  board: ArraySchema<string> = new ArraySchema<string>(null, null, ...);

  @type('string')
  currentTurn: string = 'X';

  @type('string')
  winner: string = '';

  @type(['number'])
  winLine: ArraySchema<number> = new ArraySchema<number>();
}
```

### 6.3 قواعد التحقق من رسائل الشبكة

1. رفض الرسائل من العملاء الذين ليسوا في الغرفة.
2. رفض الحركات عندما لا تكون دور المرسل.
3. رفض الفهارس خارج الحدود أو الإحداثيات غير الصالحة.
4. رفض الرسائل المكررة بأرقام تسلسلية قديمة.
5. لا تثق أبدًا بالعميل فيما يتعلق بالنتيجة أو الصحة أو حالة الفوز.

## 7. إدارة الحالة

### 7.1 حالة مستوى التطبيق

استخدم المتاجر التفاعلية للإطار:

- **Svelte:** `writable` / `readable` stores
- **Vue:** `ref` / `reactive` / Pinia

خزن فقط حالة واجهة المستخدم العالمية: المستخدم الحالي، اللعبة الحالية، إعدادات الصوت، حالة الاتصال.

### 7.2 حالة مستوى اللعبة

احتفظ بحالة اللعبة داخل مشهد Phaser. اكشف فقط ما يحتاجه الهيكل من خلال الأحداث:

```ts
this.events.emit('score', { value: 120 });
this.events.emit('gameOver', { winner: 'left' });
```

### 7.3 حالة الخادم

مخططات Colyseus هي المصدر الموثوق للألعاب الجماعية. تعكس العملاء حالة الخادم وتتنبأ/تستكمل فقط من أجل سلاسة المرئية.

## 8. سلامة الأطفال والأمان

### 8.1 التواصل

- **لا توجد دردشة نصية حرة.** تفاعلات إيموجي/عبارات جاهزة فقط.
- إذا تمت إضافة الصوت أو النص لاحقًا، يجب أن تكون خاضعة للإشراف وموافقة ولي الأمر.

### 8.2 غرف الألعاب الجماعية

- استخدم معرفات غرف عشوائية غير متسلسلة (مثلًا 8 أحرف أبجدية رقمية).
- نفذ حدودًا لسعة الغرفة وإخراج اللاعبين غير النشطين.
- اسمح للاعبين بالإبلاغ أو مغادرة الغرفة على الفور.

### 8.3 خصوصية البيانات

- اجمع الحد الأدنى من البيانات: اللقب (اختياري)، الصورة الرمزية، النتائج.
- لا تخزن معرفات شخصية للأطفال دون موافقة ولي الأمر.
- امتثل لوائح حماية البيانات في عُمان وCOPPA/GDPR إذا كان الخدمة خارج عُمان.

### 8.4 حمايات جانب العميل

- تحقق من جميع المدخلات (رموز الغرف، الألقاب).
- نقّح أي نص يُعرض في DOM.
- قدم المحتوى عبر HTTPS فقط.

## 9. الصوت والأصول

### 9.1 استراتيجية الصوت

- احتفظ بـ Web Audio المُولد لأصوات واجهة المستخدم (نقرات، فرقعات).
- استخدم أصولًا مضغوطة (OGG/WebM + MP3 fallback) للموسيقى والمؤثرات الصوتية الثقافية.
- قدم زر كتم الصوت الرئيسي وضوابط مستوى الصوت لكل لعبة.

### 9.2 خط أنابيب الفن

- أنشئ مكتبة أصول مشتركة: قلاع عُمانية، كثبان صحراوية، نخيل، جمال، ملابس تقليدية، إلخ.
- استخدم أطلسات الأنسجة لتحسين الأداء.
- حسّن الصور إلى WebP/AVIF حيثما كان مدعومًا.

## 10. الرسومات والرسوم المتحركة

### 10.1 الهوية البصرية والتوجه الفني

يجب أن تشعر المنصة بالأصالة العُمانية مع البقاء ودودة وقابلة للقراءة للأطفال في الفئة العمرية 7–12.

#### 10.1.1 الركائز المرئية الثقافية

- **الصحراء والكثبان:** نغمات رملية دافئة، كثبان متموجة، صخور مشمسة.
- **البحر والساحل:** مياه فيروزية، سفن شراعية، أسماك، شعاب مرجانية، ميناء مطرح.
- **الجبال والوديان الخضراء:** مصاطب جبل الأخضر، الورد، أشجار الرمان.
- **الرموز التراثية:** القلاع (نخل، جبرين، بهلا)، أشجار اللبان، جمال، خناجر (خنجر)، سيوف، فخار.
- **الاحتفالية والاجتماعية:** حلوى عُمانية، دلة قهوة، تمور، ملابس تقليدية.

#### 10.1.2 لوحة الألوان الموحدة

اعتمد اللوحة على متغيرات CSS الحالية ووسعها لفن اللعبة:

| الرمز | Hex | الاستخدام |
|-------|-----|-----------|
| `--sandy` | `#e6d5b8` | الكثبان، الخلفيات |
| `--sand-dark` | `#cbb593` | الظلال، الصخور |
| `--sea` | `#1e6f7a` | الماء، واجهة المستخدم الأساسية |
| `--sea-dark` | `#15565e` | الماء العميق، الرؤوس |
| `--sun` | `#f4b942` | اللمسات، الشمس، العملات |
| `--sun-dark` | `#d99a2b` | حالات التمرير |
| `--success` | `#27ae60` | التغذية الراجعة الإيجابية |
| `--danger` | `#c0392b` | الضرر، التحذيرات |
| `--white` | `#ffffff` | النص على الداكن، اللمعات |
| `--charcoal` | `#2f2f2f` | نص الجسم |

استخدم اللوحة باستمرار across جميع الألعاب حتى تشعر المنصة بالتماسك.

#### 10.1.3 الطباعة

- **خط واجهة المستخدم:** `Cairo` / `Tajawal` (قيد الاستخدام بالفعل).
- **نص داخل اللعبة:** استخدم نفس عائلة الخطوط؛ تجنب الخطوط الزخرفية التي تقلل من قراءة النص العربي.
- **الأحجام الدنيا:** 16px للنص، 22px للنتائج، 32px للعناوين على الجوال.
- **ارتفاع السطر:** 1.5 أو أكبر للنص العربي.

#### 10.1.4 تصميم الشخصيات

- **الأسلوب:** ودود، نسب مستديرة، عيون كبيرة، سيلويتات قابلة للقراءة.
- **الصور الرمزية:** اسمح للأطفال باختيار من مجموعة شخصيات ذات طابع ثقافي (ولد/بنت في زي عُماني، جمل، صقر، إلخ).
- **اختبار السيلويت:** يجب التعرف على كل شخصية من سيلويتها وحده.
- **تجنب:** تفاصيل معقدة للغاية لا تقرأ في الأحجام الصغيرة.

#### 10.1.5 ثيمات البيئات

| نوع اللعبة | مؤشرات البيئة |
|------------|---------------|
| Runner | كثبان وهيبة، أشجار اللبان، وادي قديم |
| Tic-Tac-Toe | لوحة خنجر ضد سيف، بلاط حجر رملي |
| Fort Battle | قلعتان عُمانيتان عبر وادي، أعلام الرياح |
| Racing | طريق ساحلي، ممر جبلي، مسار صحراوي |
| Memory | بلاط تراثي: فخار، مجوهرات، أنماط الملابس |
| Puzzle | قنوات الفلاج، أنماط عُمانية هندسية |

### 10.2 معايير الأصول

#### 10.2.1 أنواع الأصول

| الفئة | أمثلة |
|-------|-------|
| Sprites | شخصيات اللاعب، الأعداء، العناصر القابلة للجمع، المقذوفات |
| Backgrounds | طبقات parallax، مشاهد ثابتة، تدرجات السماء |
| UI | أزرار، لوحات، شارات، أيقونات، نوافذ منبثقة |
| Particles | غبار، شرر، كونفيتي، رذاذ ماء، نار |
| Effects | فلاش، اهتزاز الشاشة، vignette، انتقالات |
| Fonts | خطوط Cairo/Tajawal، خطوط النتائج الرقمية |

#### 10.2.2 تنسيقات الملفات

| نوع الأصل | التنسيق المفضل | البديل |
|-----------|----------------|--------|
| الصور الثابتة | WebP | PNG |
| Sprites بشفافية | WebP (lossless) | PNG-8 |
| أطلسات الأنسجة | PNG + JSON | — |
| الصوت | OGG | MP3 |
| بيانات الرسوم المتحركة | JSON (Phaser atlas/Spine/DragonBones) | — |
| واجهة المستخدم المتجهة | SVG | — |

#### 10.2.3 الدقة والأحجام

- **دقة اللوحة الداخلية:** استهدف 1920×1080 كحد أقصى؛ قلل للأداء على الأجهزة اللوحية.
- **أحجام Sprite:** صمم بحجم 2× أو 3× حجم العرض في اللعبة، ثم قلل الحجم.
- **الحد الأقصى لحجم الأنسجة:** 2048×2048 لتوافق واسع للأجهزة.
- **ميزانية الأصول لكل لعبة:** استهدف < 5 ميجابايت إجمالي (صور + صوت).

- استخدم أدوات مثل **TexturePacker** أو **Shoebox** أو **أداة إنشاء أطلس Phaser 3 المدمجة**.
- احتفظ بأطلس واجهة المستخدم منفصلًا عن أطلس اللعبة لإدارة الذاكرة.

### 10.3 مبادئ الرسوم المتحركة

#### 10.3.1 التوقيت للأطفال

- **الإجراءات:** 0.15–0.3 ثانية لتغذية راجعة واجهة المستخدم.
- **الانتقالات:** 0.3–0.5 ثانية لتغييرات الشاشة/النافذة المنبثقة.
- **الاستعداد:** استخدم 2–4 إطارات من الرياح قبل الإجراءات الكبيرة (قفز، إطلاق).
- **التوقف على الفوز:** احتفل لمدة 1.5–2 ثانية حتى يلاحظ الأطفال الفوز.
- **تجنب:** الوميض، الوميض السريع، أو الحركات التي ق تسبب إزعاجًا.

#### 10.3.2 رسوم متحركة للتغذية الراجعة

كل تفاعل يجب أن يوفر تغذية راجعة بصرية فورية:

| التفاعل | التغذية الراجعة |
|---------|-----------------|
| ضغط الزر | تصغير 0.95 + زيادة السطوع |
| اختيار الخلية | ظهور منبثق + توهج خفيف |
| الإجراء الصحيح | فلاش أخضر + ارتفاع النتيجة |
| الإجراء الخاطئ | اهتزاز أحمر + صوت قصير |
| تلقي الضرر | اهتزاز الشاشة + نبض vignette |
| الجمع | انكماش العنصر في عداد النتيجة |
| الفوز | كونفيتي + احتفال الشخصية |
| الخسارة | تلاشٍ لطيف + مطالبة إعادة المحاولة |

#### 10.3.3 حالات الرسوم المتحركة للشخصيات

يجب أن تدعم معظم شخصيات اللاعب:

- **Idle:** تنفس/حركة خفيفة متكررة.
- **Run:** دورة 6–10 إطارات.
- **Jump:** استعداد → انطلاق → قمة → هبوط.
- **Duck/Crouch:** وضعية مضغوطة، انتقال سريع.
- **Attack/Shoot:** رياح → إجراء → استرداد.
- **Celebrate:** رفع الذراعين، ارتداد، ابتسامة.
- **Hurt:** فلاش أحمر قصير، وضعية تمايل.

#### 10.3.4 الانتقالات

- استخدم انتقالات التلاشي أو الانزلاق بين هيكل التطبيق ومشاهد اللعبة.
- حافظ على الانتقالات أقل من 0.4 ثانية للحفاظ على المشاركة.
- تجنب انتقالات الصفحات المعقدة للغاية على الأجهزة اللوحية منخفضة المواصفات.

#### 10.3.5 الجاذبية والمبالغة

- استخدم **squash and stretch** على الشخصيات وعناصر واجهة المستخدم.
- أضف **follow-through** (مثل الوشاح/الشعر يتدلى على الrunner).
- استخدم **الحركة الثانوية** (أعلام ترفرف، أوراق نخيل تتمايل) لإحياء المشاهد.

### 10.4 التنفيذ التقني

#### 10.4.1 أدوات الرسوم المتحركة في Phaser 3

- **Tweens:** لتغذية راجعة واجهة المستخدم، عدادات النتائج، انتقالات النوافذ المنبثقة.
- **Animations:** لدورات الشخصيات المبنية على sprite.
- **Particles:** للكونفيتي، الغبار، الشرر، الماء.
- **Camera effects:** اهتزاز، تلاشٍ، فلاش، تقريب.
- **Timers:** للرسوم المتحركة المؤجلة وتنظيم التسلسل.

مثال على tween لارتفاع النتيجة:

```ts
this.tweens.add({
  targets: scoreText,
  scale: { from: 1, to: 1.4 },
  y: '-=30',
  alpha: { from: 1, to: 0 },
  duration: 800,
  ease: 'Back.easeOut',
});
```

#### 10.4.2 الرسوم المتحركة الهيكلية

للشخصيات المعقدة (مثل جمل يركض، صورة رمزية ترقص)، فكر في:

- **Spine** (مدفوع، معيار الصناعة).
- **DragonBones** (مجاني، مفتوح المصدر).
- صدّر إلى JSON + atlas وحمّل في Phaser عبر الإضافات.

استخدم الرسوم المتحركة الهيكلية فقط عندما تكون الإطار-بإطار مكلفة أو صلبة.

#### 10.4.3 مؤثرات الجسيمات

| التأثير | حالة الاستخدام |
|---------|----------------|
| كونفيتي | احتفال بالفوز |
| غبار | الجري/الهبوط |
| شرر | سهم يصدم الحجر |
| رذاذ | ألعاب الماء/الصيد |
| وهج النار | فرن الفخار |
| أوراق | ألعاب الرياح في الجبال |

حافظ على عدد الجسيمات معقولًا:
- الجوال: < 200 جسيم نشط.
- سطح المكتب: < 500 جسيم نشط.

#### 10.4.4 Parallax والطبقات

- استخدم 2–4 طبقات parallax لألعاب runner/explorer.
- الطبقات الأقرب إلى الكاميرا تتحرك أسرع من الطبقات البعيدة.
- خزن الطبقات الثابتة كنسيج واحد عندما يكون ذلك ممكنًا.

#### 10.4.5 قواعد تأثيرات الشاشة

| التأثير | متى يستخدم | الحد |
|---------|------------|------|
| اهتزاز الشاشة | ضرر ثقيل، انهيار القلعة | < 0.3 ثانية، شدة منخفضة |
| فلاش | ضربة ناجحة، عنصر قوة | 50–100 مللي ثانية، لون ناعم |
| Vignette | صحة منخفضة، خطر | نبض، ليس ثابتًا |
| تقريب | التركيز على الفوز/لحظة مهمة | موجز، خفيف |
| تلاشٍ | انتقالات المشاهد | 0.3–0.5 ثانية |

#### 10.5.1 ميزانيات الأداء

| المقياس | الهدف |

#### 10.5.2 قواعد التحسين

- أعد استخدام الأنسجة عبر الألعاب حيثما كان ذلك مناسبًا ثيميًا.
- تجمع الجسيمات والsprites بدلاً من إنشائها/تدميرها باستمرار.

#### 10.5.3 إمكانية الوصول

- احترم `prefers-reduced-motion`: عطّل اهتزاز الشاشة، قلل عدد الجسيمات، بسّط الانتقالات.
- حافظ على نسب تباين الألوان ≥ 4.5:1 للنص وواجهة المستخدم المهمة.
- لا تعتمد فقط على اللون لنقل حالة اللعبة؛ استخدم الأيقونات/الأشكال أيضًا.
- قدم مؤشرات تركيز بصرية واضحة للتنقل بلوحة المفاتيح.

### 16.4 دعم العمل دون اتصال وعمليات PWA

#### 16.4.1 القدرة دون اتصال حسب نوع اللعبة

- قدم مطالبة "إعادة التحميل للتحديث" عند توفر إصدار جديد.

### 16.5 معالجة الأخطاء والإبلاغ عن الأعطال

#### 16.5.1 حدود الخطأ العالمية

- امسك الأخطاء غير المعالجة في هيكل التطبيق.
- امسك الأخطاء داخل مشاهد Phaser وعد إلى الشاشة الرئيسية بأمان.
- لا تعرض آثار المكدس للأطفال.

#### 16.5.2 التدهور السلس

| الفشل | السلوك |
|-------|--------|
| WebGL غير متوفر | عرض رسالة ودية "حدّث متصفحك" |
| سياق الصوت محظور | الاستمرار بصمت؛ لا تعطل |
| WebSocket منقطع | عرض spinner إعادة الاتصال؛ السماح بالانضمام مرة أخرى |
| فشل تحميل أصل | إعادة المحاولة مرة واحدة، ثم عرض بديل أو تخطي |

#### 16.5.3 الإبلاغ عن الأعطال

- أرسل الأخطاء إلى Sentry أو نقطة نهاية مخصصة.
- ضمّن معرف اللعبة، المشهد، نوع الجهاز، وإصدار المتصفح.
- امسح أي معلومات تعريف شخصية.

#### 17.6.3 قواعد الأمان للغرف الأكبر

- **حدود السعة:** سقف صارم عند 6 لاعبين لـ FFA، و6 لاعبين للفرق.
- **ضوابط المضيف:** المضيف يمكنه طرد اللاعبين الخاملين أو المزعجين.
- **لا دردشة نصية:** تفاعلات الإيموجي فقط تظل إلزامية.

- **وضع مشرف:** وضع الفصل الدراسي هو السياق الوحيد المسموح فيه بالغرف الأكبر بدون دعوات.

#### 17.6.4 اعتبارات تقنية

- غرف Colyseus تدعم عدة عملاء، لكن المخططات ومنطق اللعبة يجب إعادة تصميمهما.
- ترتيب الدور يصبح قائمة انتظار أو تناوب قائم على الفريق بدلاً من تبديل بسيط X/O.
- شروط الفوز/الخسارة يجب أن تكون مدركة للفريق أو الترتيب.
- حجم مزامنة الحالة يزداد مع عدد اللاعبين؛ حسّن مبكرًا.
- ترحيل المضيف: إذا غادر المضيف، يجب على الخادم أو لاعب آخر تولي المهمة.
